PART 1 — WHAT WE ARE BUILDING
================================================================================

--------------------------------------------------------------------------------
1.1 THE PROGRAM
--------------------------------------------------------------------------------

GripTape runs a passion-based learning program for NYC teenagers ages 15-19.
Youth apply, get accepted, receive a $250 learning stipend, get matched with
a mentor (Champion), meet with them bi-weekly over 8 weeks, and present their
work at a Grand Showcase event.

This platform automates the entire journey — from application screening to
certificate issuance — so 1-3 staff can run 1,000 youth simultaneously with
minimal daily effort.

The MVP is designed for a pilot of 50 youth. Everything is built lean.
Complexity is added in V2 when the pilot proves the program works.


--------------------------------------------------------------------------------
1.2 DESIGN PRINCIPLES
--------------------------------------------------------------------------------

- One staff member must be able to oversee 1,000 youth with minimal effort.
  The system handles routine work. Humans handle exceptions only.

- No cohort cycles. Youth enter on a rolling basis. One kid can be in
  Week 1 while another is finishing Week 8.

- State machine. Every youth record has a single status field that moves
  forward through defined states. The system always knows where every
  youth is and what happens next.

- Build lean for the pilot. Every architectural decision is tested against:
  "Does this solve a problem I have right now with 50 youth?" If the answer
  is "no, but it will matter at 1,000" — add the column, skip the logic.
  The column costs nothing. The logic costs days.

- Configuration over code. Copy, deadlines, AI prompts, and email templates
  never live inside logic files. Change the program without touching code.

- Supabase is the only source of truth. Airtable is a read-only mirror
  for staff reporting. Nothing ever writes to Airtable directly.


--------------------------------------------------------------------------------
1.3 THE FULL YOUTH JOURNEY
--------------------------------------------------------------------------------

================================
STAGE 1 — APPLICATION  [LIVE]
================================

What happens:
  Youth fills out the chat-style application form on the GripTape website.

What they submit:
  Name, age, borough/neighborhood, school, passion (40+ words required),
  why they want to join, email and phone number.

AI check:
  Claude checks three criteria:
    1. Age is 15-19
    2. Location is an NYC borough or New Jersey
    3. Passion answer is real and specific (not gibberish)

If all pass:
  Status -> video_pending.
  Email + SMS sent inviting them to submit an intro video.
  10-day deadline starts. Magic link included in message.
  A profile_token is generated and stored on the applications row.
  A profile link is included in the video_pending email:
    BASE_URL/profile?token=[profile_token]

If any fail:
  Status -> rejected.
  Kind rejection email + SMS sent immediately.

If unclear:
  Status -> flagged.
  No message to youth.
  Staff notified via SMS immediately.

Tech:
  Existing HTML/JS form on Vercel.
  Submits to /api/submit.
  Writes to Supabase. Supabase webhook syncs to Airtable.


  BUILDER PROFILE PAGE  [LIVE]
  ----------------------------
  Every accepted applicant gets a personal profile page at:
    BASE_URL/profile?token=[profile_token]

  The profile_token never expires and is read-only — it gates display
  only, never submission. The page is designed to be shared with friends
  and family.

  The page renders the youth's passion and why_join answers, their
  member-since date, and a status badge. Locked sections
  (Declaration, First Drop, Champion) are visually present but inactive —
  they will surface links to the relevant submission forms as the youth
  progresses through the program. Submissions always happen on dedicated
  form pages — the profile page is display only.

  Archetype set (for future Claude-generated trading card — not yet built):
    The Builder   #C8F135    The Narrator  #7B2FFF
    The Sonic     #38CFFF    The Visionary #FF5C5C
    The Maker     #FFB840    The Activist  #FF6B6B
    The Connector #A8D020    The Pioneer   #9B5FFF


================================================
STAGE 2 — INTRO VIDEO SUBMISSION  [BUILD NEXT]
================================================

What happens:
  Youth submits a YouTube link via their magic link form.

Deadline:
  10 days from acceptance message.
  Nudge SMS sent at day 7 if no submission.

If deadline missed:
  Status -> removed. Youth notified by email + SMS.

AI check:
  Claude verifies the YouTube link is real, public, and watchable.
  Uses YouTube oEmbed API to confirm.

If video valid:
  Status -> onboarding.
  Confirmation email + SMS sent. Champion matching begins.

If video invalid:
  Status -> rejected. Youth notified. One chance to resubmit in 48 hours.

Tech:
  forms/video/ — simple form, token-gated via magic link.
  Supabase Edge Function on row update.
  YouTube oEmbed API (free, no key required).


====================================================
STAGE 3 — CHAMPION MATCHING & MEETING  [BUILD NEXT]
====================================================

What happens:
  System finds an available Champion and matches them to the youth.

Matching logic:
  Query champions where availability = true and
  active_youth_count < max_youth. Pass available Champions' bios
  and youth's passion description to Claude. Claude picks the
  best soft match. If no Champions available, staff is alerted.

Notifications:
  Introduction email + SMS sent to both youth and Champion.
  Includes each other's name, bio, and instructions to meet
  within the deadline.

Meeting deadline:
  Set in config.js (default: 14 days).
  Youth submits meeting confirmation form by the deadline.

Meeting form:
  Confirms meeting happened.
  Includes a short reflection from the youth.
  Champion confirms via SMS reply (YES to confirm).

If deadline missed:
  Status -> removed. Youth notified. Champion's active_youth_count
  decremented.

If form submitted:
  Status -> grant_pending. W9 + grant request forms sent to youth.

Tech:
  Supabase Edge Function (matching).
  forms/meeting/ — token-gated form for youth reflection.
  Champion confirms via inbound SMS reply.
  Resend (email). Twilio (SMS).


===============================================
STAGE 4 — GRANT REQUEST (W9 + FUNDS)  [BUILD NEXT]
===============================================

What happens:
  Youth receives W9 and grant request forms to complete so GripTape
  can legally send the $250 learning stipend.

Deadline:
  Set in config.js (default: 14 days). Nudge at day 10.

If deadline missed:
  Status -> removed. Youth notified.

AI review:
  Claude checks W9 is complete and grant request is properly filled.
  Flags any missing fields.

Staff approval:
  Staff receives SMS: AI review summary + approve/reject prompt.
  Staff replies to approve. System triggers fund disbursement.
  Status -> active.

Tech:
  forms/grant/ — token-gated.
  Supabase Edge Function for AI review.
  Twilio SMS to staff for approval.
  Payment method TBD (confirm with accountant before building).


==========================================
STAGE 5 — ACTIVE PROGRAM (8 WEEKS)  [FUTURE]
==========================================

What happens:
  Youth is active. Bi-weekly Champion meetings. Weekly email updates.

Bi-weekly nudges:
  SMS sent to youth and Champion before each meeting week (2, 4, 6, 8).

Meeting confirmation:
  Same form as Stage 3. Youth confirms + submits reflection.
  If missed, staff is flagged. Youth is NOT auto-removed at this stage.

Weekly emails:
  All active youth receive a weekly group email — project updates,
  motivational messages, program news. Staff drafts content. Resend sends.

Week 8 trigger:
  Email + SMS sent prompting youth to submit their final video.

Tech:
  Supabase scheduled functions.
  Resend (weekly emails). Twilio (SMS nudges).


=========================================
STAGE 6 — FINAL VIDEO & SHOWCASE  [FUTURE]
=========================================

What happens:
  Youth submits a final YouTube video showcasing their passion project.

Deadline:
  Set in config.js. Nudge SMS at day 5 if no submission.

AI check:
  Same as Stage 2 — real, public, watchable video.

If valid:
  Status -> showcase_invited.
  Congratulations email + SMS sent with Showcase event details + RSVP link.

If invalid or missed:
  Staff flagged. No auto-remove. Staff decides next step.

Showcase RSVP:
  forms/rsvp/ — simple form. Stored in event_rsvps table.

Completion:
  After Showcase, status -> completed. Certificate emailed.

Alumni:
  Completed youth optionally invited to become Champions.
  Record flagged in Airtable for outreach.

Tech:
  Supabase. Resend. Twilio. YouTube oEmbed API.

--------------------------------------------------------------------------------
2.1 THE STATUS FIELD
--------------------------------------------------------------------------------

Every youth record has a `status` text field. This is the heart of the system.
All Edge Functions read and write it. Airtable mirrors it. Nothing moves
without it.

Valid status values (defined in config.js — not a database enum):

  submitted           Application received, awaiting AI screening
  screening           AI checking eligibility
  video_pending       Awaiting intro video
  video_review        Video submitted, AI checking
  rejected            Did not meet criteria. TERMINAL.
  onboarding          Accepted, Champion matching in progress
  mentor_pending      Matched, awaiting meeting confirmation
  grant_pending       Meeting confirmed, awaiting W9 + grant forms
  grant_review        Forms submitted, AI reviewed, awaiting staff approval
  active              Funded, in program, bi-weekly meetings
  final_video_pending Awaiting final video
  showcase_invited    Invited to Grand Showcase
  completed           Program complete. TERMINAL.
  removed             Missed deadline or removed. TERMINAL.

RULES:
  - Status only moves forward. Never backward except via staff override.
  - Terminal states: rejected, removed, completed. No automations fire
    after the terminal notification is sent.
  - Every status change writes a timestamp field for audit trail.
  - Every status change triggers the Airtable sync webhook.
  - Status values are treated as permanent. Never rename them.
    Adding new values is fine. Renaming cascades everywhere.

2.4 CONFIG.JS CONVENTIONS
--------------------------------------------------------------------------------

config.js is the control panel for the program. All deadline values,
stage definitions, and program parameters live here. Nothing is hardcoded
inside Edge Functions or API routes.

Structure:

export const config = {

  PROGRAM_ID: 'nyc-2026',   // used as program_id on all records

  STAGES: {
    submitted:           { next: 'screening',           deadline_days: null, nudge_day: null,  auto_remove: false },
    screening:           { next: 'video_pending',        deadline_days: null, nudge_day: null,  auto_remove: false },
    video_pending:       { next: 'video_review',         deadline_days: 10,   nudge_day: 7,     auto_remove: true  },
    video_review:        { next: 'onboarding',           deadline_days: null, nudge_day: null,  auto_remove: false },
    onboarding:          { next: 'mentor_pending',       deadline_days: null, nudge_day: null,  auto_remove: false },
    mentor_pending:      { next: 'grant_pending',        deadline_days: 14,   nudge_day: 10,    auto_remove: true  },
    grant_pending:       { next: 'grant_review',         deadline_days: 14,   nudge_day: 10,    auto_remove: true  },
    grant_review:        { next: 'active',               deadline_days: null, nudge_day: null,  auto_remove: false },
    active:              { next: 'final_video_pending',  deadline_days: null, nudge_day: null,  auto_remove: false },
    final_video_pending: { next: 'showcase_invited',     deadline_days: 14,   nudge_day: 5,     auto_remove: false },
    showcase_invited:    { next: 'completed',            deadline_days: null, nudge_day: null,  auto_remove: false },
  },

  TERMINAL_STAGES: ['completed', 'rejected', 'removed'],

  STIPEND_AMOUNT: 250,
  PROGRAM_DURATION_WEEKS: 8,
  MEETING_FREQUENCY_WEEKS: 2,
  DEFAULT_CHAMPION_CAPACITY: 3,
  TOKEN_BUFFER_HOURS: 24,   // extra time before token expires after deadline

  ELIGIBLE_LOCATIONS: ['NYC', 'NJ'],
  AGE_MIN: 15,
  AGE_MAX: 19,
  PASSION_MIN_WORDS: 40,

}

RULES:
  - Edge Functions import config and read from it. Never hardcode values.
  - Changing a deadline = update one line in config.js, push to GitHub.
  - Adding a new stage = add a row to STAGES, add a block in content.js,
    add an email template, write the Edge Function. No schema changes needed.

--------------------------------------------------------------------------------
2.6 PROMPT FILE CONVENTIONS
--------------------------------------------------------------------------------

Every Claude API call has its prompt in prompts/. Never write prompts
inline inside Edge Functions.

Each file exports:
  systemPrompt          instruction set for Claude
  buildUserPrompt(data) function that returns the user message
  outputSchema          expected JSON response structure

Example — prompts/screenApplication.js:

  export const systemPrompt = `
    You are screening applications for the GripTape Learning Challenge.
    Evaluate the application against the eligibility criteria provided.
    Respond only in valid JSON. Be encouraging. This is for NYC teenagers.
  `;

  export const buildUserPrompt = (application, config) => `
    Eligibility criteria:
    - Age: ${config.AGE_MIN} to ${config.AGE_MAX}
    - Locations: ${config.ELIGIBLE_LOCATIONS.join(', ')}
    - Passion: minimum ${config.PASSION_MIN_WORDS} words, real and specific

    Application:
    Name: ${application.name}
    Age: ${application.age}
    Location: ${application.borough}
    Passion: ${application.passion}
    Why join: ${application.why_join}
  `;

  export const outputSchema = {
    decision: "accepted | rejected | flagged",
    reasoning: "string",
    failed_criteria: "string | null"
  };

RULES:
  - Always use model: claude-sonnet-4-20250514
  - Always request JSON output in the system prompt
  - Never hardcode eligibility criteria — always read from config
  - Output schema always includes: decision, reasoning, failed_criteria


--------------------------------------------------------------------------------
2.7 MAGIC LINK SYSTEM
--------------------------------------------------------------------------------

Youth access all forms via unique magic links. No login, no password.
Champions receive a unique registration link the same way.

HOW IT WORKS:
  1. Youth advances to a stage that requires a form submission
  2. System generates a unique token via lib/tokens.js
  3. Token stored on youth record: access_token + token_expires_at
  4. token_expires_at = stage entered timestamp + stage deadline + buffer
  5. Link sent via SMS + email: https://griptape.org/video?token=abc123
  6. Form validates token on load via Supabase lookup
  7. Valid: form loads with youth name pre-filled
  8. Expired or invalid: youth sees "get a new link" screen,
     enters email, system sends a fresh link

TOKEN RULES:
  - Tokens expire when the stage deadline passes (not on first use)
  - Youth can return to the same form multiple times before deadline
  - lib/tokens.js handles all generation and validation
  - Never implement token logic directly in a form or Edge Function

CHAMPION REGISTRATION:
  - Staff generates a link: texts staff agent number with
    "Add Champion — send link to [phone]"
  - System creates a pending champion record + registration_token
  - Champion opens forms/champion-register/, fills profile, sets capacity
  - On submit: availability = true, registered_at = now
  - Staff notified via SMS: "[Name] has registered as a Champion"


--------------------------------------------------------------------------------
2.8 AIRTABLE SYNC
--------------------------------------------------------------------------------

Airtable is a read-only mirror of Supabase for staff reporting.
Nothing writes to Airtable directly. Ever.

HOW IT WORKS:
  - Every Supabase record change fires a webhook to api/sync-airtable.js
  - sync-airtable.js writes the change to Airtable via the Airtable API
  - If a record doesn't exist in Airtable yet, it is created
  - One-way only: Supabase -> Airtable. Never the reverse.

TABLES MIRRORED:
  - applications
  - youth
  - champions
  - grant_requests
  - meetings

FIELD MAPPING:
  Supabase uses snake_case. Airtable uses Title Case With Spaces.
  Mapping lives in api/sync-airtable.js as a fieldMap object.
  Adding a new Supabase column = add Airtable column + update fieldMap.

RULES:
  - If Airtable is down, events are lost for that window (acceptable
    for MVP — add a sync queue in V2 if this becomes a problem)
  - Supabase is always the source of truth
  - Staff reads Airtable, acts via SMS agent or direct Supabase access


--------------------------------------------------------------------------------
2.9 STAFF NOTIFICATIONS (MVP)
--------------------------------------------------------------------------------

The full staff AI agent is a V2 feature. For the MVP, staff are notified
via simple Twilio SMS for the cases that require human action.

STAFF IS NOTIFIED WHEN:
  - An application is flagged (AI unclear on eligibility)
  - No Champions are available for a youth awaiting matching
  - A grant request is ready for approval (includes AI review summary)
  - A final video is flagged or deadline missed (staff decides next step)
  - An Edge Function throws an unhandled error

GRANT APPROVAL FLOW (MVP):
  1. Youth submits W9 + grant request
  2. AI reviews, writes notes to grant_requests.ai_review_notes
  3. Staff receives SMS: youth name + AI review summary + "Reply APPROVE
     or DECLINE"
  4. Staff replies APPROVE
  5. System updates staff_approved = true, triggers fund disbursement
  6. Status -> active

ALL STAFF NOTIFICATIONS:
  - Logged to agent_log table with triggered_by = 'system'
  - Include youth name, current status, and clear next action
  - Come from a dedicated Twilio number (separate from youth SMS)


--------------------------------------------------------------------------------
2.10 STATE TRANSITION PATTERN
--------------------------------------------------------------------------------

MANDATORY PATTERN — Webhook-triggered Edge Functions
-----------------------------------------------------
Used when a Supabase database change (not a form submission) should trigger
an Edge Function, or multiple downstream functions might react to one event.

  1. FORM (Vercel HTML) — submits to Vercel API route
  2. THIN VERCEL API ROUTE — validates token, writes to Supabase
  3. SUPABASE WEBHOOK — fires Edge Function on row change
  4. EDGE FUNCTION — reads webhook payload, calls advance_status() RPC first
     to claim the row atomically, skips if already claimed, runs logic

  advance_status() signature:
    rpc('advance_status', { p_id, p_expected_status, p_new_status })
    Returns true if this caller won the race, false if already claimed.

  EXAMPLE — validate-video uses webhook (video_url column change triggers it).


DIRECT INVOCATION PATTERN
-------------------------
Used when a Vercel API route controls exactly which Edge Function
fires next. Eliminates webhook duplicate fire risk for form-triggered
transitions.

  Pattern:
    1. FORM (Vercel HTML) — same as mandatory pattern
    2. THIN VERCEL API ROUTE — validates token, writes to Supabase,
       calls Edge Function directly via fetch using SUPABASE_SERVICE_KEY
    3. EDGE FUNCTION — reads application_id from POST body, fetches
       full record from Supabase, runs business logic, sends comms
    No webhook. No duplicate fire risk.

  USE THIS PATTERN WHEN:
    Form submission directly and exclusively triggers one Edge Function.

  USE WEBHOOK PATTERN WHEN:
    A data change (not a form) should trigger an Edge Function, or
    multiple downstream functions might react to one event.

  EXAMPLE — process-declaration uses direct invocation.
  EXAMPLE — validate-video uses webhook (video_url column change).

4.4 RULES FOR PROMPTING CLAUDE CODE
--------------------------------------------------------------------------------

Start every Claude Code prompt with this header:

  "Using the GripTape master spec (GripTape_Master_Spec.txt), build
  [component]. Follow all architecture conventions exactly:
  - All copy goes in content.js, organized by stage key
  - All AI prompts go in prompts/
  - All deadline and config values come from config.js
  - All email sends use lib/email.js, all SMS sends use lib/sms.js
  - Never hardcode API keys — use environment variables
  - Column names must match the schema in section 2.2 exactly
  - Status values must match the valid values in section 2.1 exactly
  - Every status change writes a timestamp field for audit trail"

ADDITIONAL RULES:
  - The status field is sacred. Only move it forward. Never skip a state.
  - Update status atomically with any other field changes in the same
    database operation.
  - All scheduled jobs must be idempotent. Running twice must not send
    duplicate notifications. Use sent_at timestamp fields to check first.
  - lib/tokens.js handles all magic link logic. Never implement token
    logic directly in a form or Edge Function.
  - Never write to Airtable from Edge Functions. All writes go to
    Supabase. Airtable receives changes via api/sync-airtable.js only.
