PART 1 — WHAT WE ARE BUILDING
================================================================================

--------------------------------------------------------------------------------
1.1 THE PROGRAM
--------------------------------------------------------------------------------

GripTape runs a passion-based learning program for NYC teenagers ages 15-19.
Youth apply, get accepted, receive a $250 learning stipend, get matched with
a mentor (Champion), meet with them bi-weekly over 10 weeks, and present their
work at a Grand Showcase event.

This platform automates the entire journey — from application screening to
certificate issuance — so 1-3 staff can run 1,000 youth simultaneously with
minimal daily effort.

The MVP is designed for a pilot of 50 youth. Everything is built lean.
Complexity is added in V2 when the pilot proves the program works.


--------------------------------------------------------------------------------
1.2 DESIGN PRINCIPLES
--------------------------------------------------------------------------------

- One staff member must be able to oversee 1,000 youth with minimal effort.
  The system handles routine work. Humans handle exceptions only.

- No cohort cycles. Youth enter on a rolling basis. One kid can be in
  Week 1 while another is finishing Week 8.

- State machine. Every youth record has a single status field that moves
  forward through defined states. The system always knows where every
  youth is and what happens next.

- Build lean for the pilot. Every architectural decision is tested against:
  "Does this solve a problem I have right now with 50 youth?" If the answer
  is "no, but it will matter at 1,000" — add the column, skip the logic.
  The column costs nothing. The logic costs days.

- Configuration over code. Copy, deadlines, AI prompts, and email templates
  never live inside logic files. Change the program without touching code.

- Supabase is the only source of truth. Airtable is a read-only mirror
  for staff reporting. Nothing ever writes to Airtable directly.


--------------------------------------------------------------------------------
1.3 THE FULL YOUTH JOURNEY
--------------------------------------------------------------------------------

================================
STAGE 1 — APPLICATION  [LIVE]
================================

What happens:
  Youth fills out the chat-style application form on the GripTape website.

What they submit:
  Name, age, address, school, passion (40+ words required),
  why they want to join, email and phone number.

AI check:
  Claude screens for: age 15-19, NYC/NJ location, passion length,
  genuine interest vs spam. Returns accepted / rejected / flagged.

If accepted:
  screening_status -> declaration_pending.
  Youth receives email + SMS with magic link to declaration form.
  10-day deadline.

If rejected:
  screening_status -> rejected. Kind rejection email sent.

If flagged:
  screening_status -> flagged. Staff notified via email. No email to youth.
  Staff manually reviews and sets status.

Tech:
  HTML/JS form on Vercel — live at learning-challenge-legacy.vercel.app
  Submits to /api/submit (Vercel serverless function).
  Writes to Supabase applications table. Supabase is source of truth.
  Airtable sync is a separate job.
  Supabase DB webhook on INSERT to applications fires screen-application
  Edge Function.

NOTE: The stage between screening acceptance and video submission is
  declaration_pending — youth must complete a declaration/profile form
  before receiving their video submission link. This is an intermediate
  step not described in the original spec; it was added during build.


================================
STAGE 2 — DECLARATION  [LIVE]
================================

What happens:
  Accepted youth complete a declaration form via their magic link.
  This confirms their commitment and collects any remaining profile info.

If completed:
  screening_status -> video_pending.
  Youth receives email + SMS with magic link to video submission form.
  10-day deadline.

Tech:
  forms/declare/ — token-gated via magic link from acceptance email.
  api/declare-submit.js — direct invocation of process-declaration
  Edge Function (not webhook-triggered).
  process-declaration advances status and sends video link.


================================================
STAGE 3 — INTRO VIDEO SUBMISSION  [LIVE]
================================================

What happens:
  Youth submits a YouTube link to a short intro video about themselves
  and their passion.

Deadline:
  10 days from declaration confirmation email.

AI check:
  validate-video confirms video_url is present and advances status to
  video_review. Staff then manually reviews video in Supabase.
  (Full oEmbed validation is a future enhancement.)

If video submitted:
  screening_status -> video_review. Staff reviews manually.

If staff accepts:
  Staff sets screening_status -> accepted on the applications table.
  on-acceptance Edge Function fires, creates youth row.
  match-champion fires on youth INSERT.

Tech:
  forms/video/ — token-gated via magic link.
  Supabase DB webhook on UPDATE to applications fires validate-video.
  validate-video advances status to video_review via advance_status().
  on-acceptance fires on UPDATE when screening_status = 'accepted'.


====================================================
STAGE 4 — CHAMPION MATCHING & MEETING  [LIVE]
====================================================

What happens:
  System finds an available Champion and matches them to the youth.

Automated path:
  match-champion fires on INSERT to youth table.
  Queries champions where available = true and
  active_youth_count < max_youth.
  Passes available Champions' bios + youth's passion to Claude.
  Claude picks best soft match.
  advance_status() moves youth onboarding -> mentor_pending,
  sets champion_id + access_token + token_expires_at.
  send-champion-intro fires via webhook, sends group intro email to
  both youth and Champion.

No-champion path:
  If no champions available: staff receives email alert with exact SQL
  to manually assign a champion once one becomes available.
  Staff runs the SQL (sets champion_id + status = mentor_pending).
  send-champion-intro webhook fires automatically on status change.
  Same intro email path — no special handling required.

Webhook trigger for intro email:
  Table: youth
  Event: UPDATE
  Condition: OLD.status = 'onboarding' AND NEW.status = 'mentor_pending'
  Fires: send-champion-intro

Meeting deadline:
  14 days. Set in config.js.

If form submitted:
  Status -> grant_pending. W9 + grant request forms sent to youth.

If deadline missed:
  Status -> removed. Youth notified. Champion's active_youth_count
  decremented.

Tech:
  match-champion — Supabase Edge Function, fires on INSERT to youth.
  send-champion-intro — Supabase Edge Function, fires on webhook
    (status transition onboarding -> mentor_pending).
  forms/meeting/ — token-gated form for youth reflection.


===============================================
STAGE 5 — GRANT REQUEST (W9 + FUNDS)  [FUTURE]
===============================================

What happens:
  Youth receives W9 and grant request forms to complete so GripTape
  can legally send the $250 learning stipend.

Deadline:
  14 days. Nudge at day 10. Set in config.js.

If deadline missed:
  Status -> removed. Youth notified.

AI review:
  Claude checks W9 is complete and grant request is properly filled.
  Flags any missing fields.

Staff approval:
  Staff receives email: AI review summary + approve/reject prompt.
  Staff approves. System triggers fund disbursement.
  Status -> active.

Tech:
  forms/grant/ — token-gated.
  Supabase Edge Function for AI review.
  Payment method TBD (confirm with accountant before building).

OPEN DECISION: Payment method for $250 stipend (check, ACH, Venmo, API?)
OPEN DECISION: W9 storage compliance — confirm with accountant/lawyer.


==========================================
STAGE 6 — ACTIVE PROGRAM (10 WEEKS)  [FUTURE]
==========================================

What happens:
  Youth is active. Bi-weekly Champion meetings. Weekly email updates.

Bi-weekly nudges:
  SMS sent to youth and Champion before each meeting week (2, 4, 6, 8).

Meeting confirmation:
  Same form as Stage 4. Youth confirms + submits reflection.
  If missed, staff is flagged. Youth is NOT auto-removed at this stage.

Weekly emails:
  All active youth receive a weekly group email — project updates,
  motivational messages, program news. Staff drafts content. Resend sends.

Week 10 trigger:
  Email + SMS sent prompting youth to submit their final video.

Tech:
  Supabase scheduled functions.
  Resend (weekly emails). Twilio (SMS nudges).


=========================================
STAGE 7 — FINAL VIDEO & SHOWCASE  [FUTURE]
=========================================

What happens:
  Youth submits a final YouTube video showcasing their passion project.

Deadline:
  Set in config.js. Nudge SMS at day 5. No auto-remove — staff decides.

If valid:
  Status -> showcase_invited.
  Congratulations email + SMS with Showcase event details + RSVP link.

If invalid or missed:
  Staff flagged. Staff decides next step.

Showcase RSVP:
  forms/rsvp/ — simple form. Stored in event_rsvps table.

Completion:
  After Showcase, status -> completed. Certificate emailed.

Alumni:
  Completed youth optionally invited to become Champions.
  Record flagged in Airtable for outreach.

Tech:
  Supabase. Resend. Twilio. YouTube oEmbed API.

PART 2 — HOW IT IS BUILT
================================================================================

--------------------------------------------------------------------------------
2.1 THE STATUS FIELD
--------------------------------------------------------------------------------

Applications use screening_status. Youth use status. These are separate
plain-text fields — never Postgres enums.

APPLICATIONS — screening_status values:
  submitted           Received, awaiting AI screening
  screening           AI check in progress (claimed by advance_status)
  declaration_pending Accepted by AI, awaiting declaration form
  video_pending       Declaration complete, awaiting video submission
  video_review        Video submitted, awaiting staff review
  rejected            Did not meet criteria. TERMINAL.
  flagged             AI unclear — staff reviews manually. Not terminal.
  accepted            Staff approved video. youth row created. TERMINAL
                      for applications table.

YOUTH — status values:
  onboarding          Youth row created, champion matching in progress
  mentor_pending      Matched with Champion, awaiting orientation call
  grant_pending       Orientation confirmed, awaiting W9 + grant docs signing
  grant_review        Docs signed, awaiting staff approval
  grant_approved      Docs approved by staff, deposit link sent to youth.
                      Sits between grant_review and active.
  active              Funded, in program, bi-weekly meetings
  final_video_pending Awaiting final video
  showcase_invited    Invited to Grand Showcase
  completed           Program complete. TERMINAL.
  removed             Missed deadline or removed. TERMINAL.

RULES:
  - Status only moves forward. Never backward except via staff override.
  - All status changes go through advance_status() RPC. No exceptions.
  - advance_status() uses optimistic locking (FOR UPDATE). If status
    doesn't match expected, raises StatusConflictError and aborts.
  - Terminal states: rejected, removed, completed, accepted (applications).
    No automations fire after terminal notification is sent.
  - Status values are permanent. Never rename them. Adding new values fine.


--------------------------------------------------------------------------------
2.2 ADVANCE_STATUS() — THE ONLY WAY TO CHANGE STATUS
--------------------------------------------------------------------------------

advance_status() is a Postgres function (security definer) that:
  1. Locks the row (FOR UPDATE)
  2. Confirms current status matches expected
  3. Updates status + updated_at + any additional_fields atomically
  4. Raises StatusConflictError if status doesn't match — caller catches
     and exits cleanly (idempotency)

Signature:
  advance_status(
    record_id               uuid,
    table_name              text,       -- 'applications' | 'youth' | 'champions'
    expected_current_status text,
    next_status             text,
    additional_fields       jsonb       -- optional extra fields to set
  ) returns jsonb

Called via: supabase.rpc('advance_status', { ... })

applications uses screening_status column.
youth and champions use status column.
Whitelisted tables only — prevents SQL injection.


--------------------------------------------------------------------------------
2.3 EDGE FUNCTIONS — BUILT AND DEPLOYED
--------------------------------------------------------------------------------

All Edge Functions run on Supabase Deno runtime.
All import shared utilities from supabase/functions/_shared/.
None hardcode API keys — all use Deno.env.get().

--

FUNCTION: screen-application
  Status: LIVE
  Trigger: Supabase DB webhook on INSERT to applications
           Name: on_application_insert
  Flow:
    1. Skips if screening_status != 'submitted'
    2. advance_status(submitted -> screening) — claims the record
    3. Calls Claude with screenApplicationSystemPrompt
    4. Parses decision: accepted | rejected | flagged
    5. advance_status(screening -> declaration_pending | rejected | flagged)
       Sets access_token + stage_deadline_at on accepted records.
       Sets profile_token on accepted records.
    6. Sends notifications via dispatcher:
       accepted  -> declaration_pending email + SMS with magic link
       rejected  -> rejection email + SMS
       flagged   -> staff email alert only (no youth notification)
    7. On error: resets status back to 'submitted' so record can retry

  Idempotency: advance_status() StatusConflictError exits cleanly.
  Retry logic: exponential backoff on Claude overloaded_error (3 attempts).

--

FUNCTION: process-declaration
  Status: LIVE
  Trigger: Direct HTTP call from api/declare-submit.js (NOT a webhook)
           One call per declaration submit — no idempotency guard needed.
  Flow:
    1. Receives application_id in request body
    2. Skips if screening_status != 'declaration_pending'
    3. Generates video token via generateToken()
    4. advance_status(declaration_pending -> video_pending)
       Sets access_token + stage_deadline_at
    5. Sends declaration_confirmed notification with video_link + profile_link

  NOTE: Direct invocation (not webhook) is intentional. Ensures exactly
  one execution per declaration. No race condition possible.

--

FUNCTION: validate-video
  Status: LIVE
  Trigger: Supabase DB webhook on UPDATE to applications
           Name: on_video_url_submitted
           Fires when video_url changes on a video_pending record
  Flow:
    1. Skips if screening_status != 'video_pending'
    2. Skips if video_url is null
    3. advance_status(video_pending -> video_review)
    4. Writes video_submitted_at timestamp

  NOTE: Does not do oEmbed validation. Advances to video_review for
  staff manual review. oEmbed check is a future enhancement.
  Staff then manually sets screening_status = 'accepted' in Supabase
  to trigger on-acceptance.

--

FUNCTION: on-acceptance
  Status: LIVE
  Trigger: Supabase DB webhook on UPDATE to applications
           Name: on_application_accepted
           Fires on all UPDATEs — function checks screening_status internally
  Flow:
    1. Skips if screening_status != 'accepted'
    2. Idempotency: skips if youth row already exists for this application_id
    3. Creates youth row with status = 'onboarding', copying fields from
       application: program_id, application_id, first_name, last_name,
       email, phone, address, birthdate, first_drop_url (video_url)
    4. Logs youth_id created

  NOTE: Creating the youth row triggers match-champion automatically
  via the INSERT webhook on the youth table.

--

FUNCTION: match-champion
  Status: LIVE
  Trigger: Supabase DB webhook on INSERT to youth
           Name: on_youth_insert
  Flow:
    1. Loads full youth record + application (for passion + why_join)
    2. Queries champions where available = true AND
       active_youth_count < max_youth
    3. If no champions available:
       - Sends staff email alert with youth details + manual assignment SQL
       - Returns 200, youth stays at 'onboarding'
    4. Calls Claude with matchChampion prompt — returns champion_id + reasoning
    5. advance_status(onboarding -> mentor_pending)
       Sets champion_id, access_token, token_expires_at
    6. Increments champion.active_youth_count
    7. Does NOT send intro email — that is owned by send-champion-intro
       (triggered by the status transition webhook)

  Idempotency: advance_status() StatusConflictError exits cleanly.

  No-champion manual resolution:
    Staff receives SQL in the alert email:
      UPDATE youth
      SET champion_id = '<CHAMPION_ID>',
          status = 'mentor_pending',
          access_token = gen_random_uuid()::text,
          token_expires_at = now() + interval '16 days',
          updated_at = now()
      WHERE id = '<YOUTH_ID>' AND status = 'onboarding';
    This triggers the on_youth_mentor_pending webhook automatically.
    No additional staff action required.

--

FUNCTION: send-champion-intro
  Status: LIVE
  Trigger: Supabase DB webhook on UPDATE to youth
           Name: on_youth_mentor_pending
           Condition: OLD.status = 'onboarding' AND NEW.status = 'mentor_pending'
           Payload: { record: NEW, old_record: OLD }
  Flow:
    1. Reads youth_id from payload.record.id
    2. Confirms record.status = 'mentor_pending' (guard against stale fires)
    3. Confirms record.champion_id is not null
    4. Loads champion record
    5. Sends group intro email to youth + champion via sendEmail()
       Uses content.mentor_pending block
    6. Logs '[send-champion-intro] intro email sent for youth [id] -> champion [id]'

  This is the SINGLE canonical trigger for the champion intro email.
  Both the automated path (match-champion) and the manual staff path
  (direct SQL update) flow through this function. No duplicate sends possible.

  Does NOT call advance_status(). Status is already mentor_pending.
  Does NOT increment active_youth_count. match-champion owns that.

--

FUNCTION: process-orientation
  Status: LIVE
  Trigger: Direct HTTP call from api/orientation-submit.js
           Fire-and-forget (not awaited). Non-fatal if it fails —
           orientation data already saved to youth record.
  Flow:
    1. Receives youth_id, challenge_topic, grant_amount, grant_format,
       legal_name in request body
    2. Skips if youth.status !== 'mentor_pending'
    3. advance_status(mentor_pending → grant_pending)
       Sets access_token + token_expires_at (14 days)
    4. Creates grant_requests row (INSERT) with youth_id, program_id
    5. UPDATEs grant_requests with challenge_topic, grant_amount,
       grant_format, legal_name
    6. Calls send-grant-docs Edge Function directly

  Idempotency: advance_status() StatusConflictError exits cleanly.

--

FUNCTION: send-grant-docs
  Status: LIVE
  Trigger: Direct HTTP call from process-orientation
  Flow:
    1. Receives youth_id
    2. Idempotency: skips if boldsign_w9_id already set on grant_requests
    3. Calls BoldSign API to create W-9 signing request
       POST /v1/template/send?templateId=BOLDSIGN_W9_TEMPLATE_ID
       disableEmails: true — all communications go through our dispatcher
    4. Calls BoldSign API to create agreement signing request
       Same pattern with BOLDSIGN_AGREEMENT_TEMPLATE_ID
    5. Fetches signing URLs via getEmbeddedSignLink per document
       linkValidTill derived from config.STAGES.grant_pending.deadline_days
    6. Updates grant_requests with boldsign_w9_id + boldsign_agreement_id
    7. Sends youth email + SMS via sendNotification('grant_pending')
       with w9_link and agreement_link as template vars

--

FUNCTION: boldsign-webhook
  Status: LIVE
  Trigger: BoldSign webhook on document completion
           No JWT verification (--no-verify-jwt)
           URL: /functions/v1/boldsign-webhook
  Flow:
    1. Filters for event.eventType === 'Completed'
    2. Extracts body.data.documentId
    3. Looks up grant_requests by boldsign_w9_id or boldsign_agreement_id
    4. Writes signed_at timestamp + BoldSign dashboard URL
       to w9_signed_at/w9_doc_url or agreement_signed_at/agreement_doc_url
    5. If both w9_signed_at and agreement_signed_at are set:
       calls process-grant-signed directly

--

FUNCTION: process-grant-signed
  Status: LIVE
  Trigger: Direct HTTP call from boldsign-webhook
  Flow:
    1. Receives youth_id
    2. Skips if youth.status !== 'grant_pending'
    3. advance_status(grant_pending → grant_review)
    4. Sends staff email via sendStaffNotification('grant_review')
       with doc URLs and approve/reject placeholder instructions

  Idempotency: advance_status() StatusConflictError exits cleanly.

--

FUNCTION: on-grant-approved
  Status: LIVE
  Trigger: Supabase DB webhook on UPDATE to grant_requests
           Name: on_grant_requests_approved
           Fires when staff_approved changes to true
  Flow:
    1. Guards using payload: old_record.staff_approved !== true AND
       record.staff_approved === true (prevents duplicate fires)
    2. Sets staff_approved_at on grant_requests
    3. Skips if youth.status !== 'grant_review'
    4. advance_status(grant_review → grant_approved)
    5. Sends youth email + SMS via sendNotification('grant_approved')
       with deposit_link: 'PLACEHOLDER_DEPOSIT_LINK'
       (Tremendous integration pending — see Open Decisions)

  NOTE: Uses payload guard pattern (not DB re-read). See Section 2.9.

--------------------------------------------------------------------------------
2.5 SHARED UTILITIES (_shared/)
--------------------------------------------------------------------------------

supabase/functions/_shared/ contains shared modules imported by all Edge Functions.

  config.ts       Program config + stage definitions. Mirrors config.js for Deno.
  content.ts      All copy organized by stage key. renderContent() helper.
  dispatcher.ts   sendNotification() and sendStaffNotification() — routes to
                  email.ts and/or sms.ts based on stage config.
                  sendStaffNotification() supports both email and SMS:
                  email fires if content block has staff_email_subject +
                  staff_email_body keys; SMS fires if staff_sms key present.
                  Both are optional — missing keys skip silently.
  email.ts        Resend wrapper. sendEmail({ to, subject, html }).
  sms.ts          Twilio wrapper. sendSMS({ to, body }).
  tokens.ts       generateToken(deadline_days) — returns { access_token,
                  stage_deadline_at }. Token never expires on first use,
                  only at stage deadline.
  prompts.ts      All Claude AI prompts. One export per use case.

RULES:
  - Edge Functions import from _shared/. Never duplicate logic.
  - Copy lives in content.ts only. Never in Edge Functions.
  - All prompts live in prompts.ts. Never inline in Edge Functions.

2.9 CONCURRENCY & IDEMPOTENCY
--------------------------------------------------------------------------------

Three sources of writes can collide: Edge Functions (webhook-triggered),
daily-scheduler (cron), and inbound handlers. advance_status() prevents
all race conditions via optimistic locking.

RULE: Every status update goes through advance_status().
  If two processes try to advance the same record simultaneously, the
  second finds the status already changed and raises StatusConflictError.
  Caller catches it, logs, and exits cleanly. Record is never corrupted.

RULE: All scheduled jobs are idempotent.
  Before sending any nudge or removal notice, scheduler checks sent_at
  timestamp. If already set, skips silently. Running twice = zero duplicate sends.

RULE: Webhook functions skip on status mismatch.
  Every Edge Function checks status at the top and returns 200 if the
  record is not in the expected state. Stale webhook fires are harmless.

RULE: UPDATE webhook functions must guard using payload, not DB reads.
  Use payload.old_record and payload.record to determine what transition
  occurred. Never re-read live DB state to make the skip decision — the
  DB may have already advanced by the time the function runs, causing the
  live read to return the new status and the guard to pass incorrectly.

  Safe pattern:
    if (payload.old_record.status !== 'expected_old' ||
        payload.record.status !== 'expected_new') {
      console.log(`[fn] skipping — transition was ${oldStatus} → ${newStatus}`)
      return new Response('skipped', { status: 200 })
    }

  INSERT webhook functions are naturally safe — they fire once per row.
  This rule applies only to UPDATE-triggered functions.

CONCURRENCY FAILURE MODE:
  If advance_status() throws StatusConflictError:
    1. Log to agent_log
    2. Send staff alert
    3. Take no further action


--------------------------------------------------------------------------------
2.10 AIRTABLE SYNC
--------------------------------------------------------------------------------

Airtable is a read-only mirror of Supabase for staff reporting.
Nothing writes to Airtable directly. Ever.

HOW IT WORKS:
  Supabase record changes fire a webhook to api/sync-airtable.js.
  sync-airtable.js writes to Airtable via Airtable API.
  One-way: Supabase -> Airtable. Never the reverse.

TABLES MIRRORED: applications, youth, champions, grant_requests, meetings

RULES:
  If Airtable is down, events are lost for that window (acceptable for
  MVP — add sync queue in V2 at 500+ youth).
  Supabase is always source of truth.

PART 4.6 — KEY ARCHITECTURAL DECISIONS MADE DURING BUILD
================================================================================

These were not in the original spec but are now system rules.
Follow them in all future Edge Function work.

--

1. UPDATE webhook functions must guard using payload, not DB reads.
   Use payload.old_record and payload.record to determine what transition
   occurred. Never re-read live DB state for the skip decision — the DB
   may have already advanced by the time the function runs, causing the
   live read to return the new status and the guard to pass incorrectly.

   Safe pattern:
     if (payload.old_record.staff_approved === true ||
         payload.record.staff_approved !== true) return 200 (skipped)

   INSERT webhook functions are naturally safe — they fire once per row.
   This rule applies only to UPDATE-triggered functions.

--

2. Direct invocation vs webhook trigger.
   Direct HTTP invocation is correct for forms making multiple writes
   to the same table (prevents webhook duplicate fires). Forms that
   write to a table and need exactly one downstream execution use direct
   invocation. Webhook pattern is correct for single clean field writes.

   Examples:
     orientation-submit.js writes to youth, then calls process-orientation
     directly (fire-and-forget). NOT webhook-triggered.
     boldsign-webhook writes to grant_requests, then calls process-grant-signed
     directly. NOT a DB webhook.

--

3. BoldSign email suppression.
   disableEmails: true on all BoldSign API calls. All communications to
   youth go through our dispatcher (Resend + Twilio). BoldSign is signing
   infrastructure only — it never sends email directly to participants.

--

4. Signing URL retrieval.
   BoldSign template/send response does NOT include the signing URL.
   Must call getEmbeddedSignLink separately per document after getting
   the documentId from the send response. linkValidTill is derived from
   config.STAGES.grant_pending.deadline_days.

--

5. grant_requests row creation.
   Created by process-orientation atomically with the grant_pending status
   transition. Never created elsewhere. Fields are written in two steps:
   INSERT (program_id + youth_id), then UPDATE (challenge_topic, grant_amount,
   grant_format, legal_name) — because the INSERT fires before we have
   confirmed the status advance succeeded.

--

6. Staff approval flow (pilot).
   Staff (Thea) reviews BoldSign docs manually and sets staff_approved = true
   directly in Supabase. on-grant-approved webhook handles the rest
   automatically. No approve/reject API routes needed for the 50-youth pilot.
   This is intentional — build the manual staff path first, add the staff
   UI when volume demands it.
